# Roadmap выполнения: «Ферма невидимых кроликов»

Этот roadmap делит ТЗ на **testable этапы**. PM режет задачи **внутри одного этапа**, не смешивая соседние этапы без явной причины.

Принцип:

- каждый этап даёт наблюдаемый результат
- после каждого этапа можно руками проверить, что проект реально продвинулся
- PM режет задачи так, чтобы acceptance tests этапа закрывались без “прыжков через ступеньки”
- как только PM поднимает этап, он сразу открывает draft PR `stage_<stage>_issue_<id>_test -> preprod`, чтобы тесты и план этапа жили в GitHub, а не в голове

## Как PM этим пользуется

- Сначала выбирает **только один этап**.
- Внутри этапа режет его на 3-7 обычных задач.
- Каждая задача закрывает один вертикальный срез внутри этапа.
- Acceptance tests этапа не должны зависеть от ещё не сделанного следующего этапа.

## Stage 1. Базовый каркас и карта фермы

### Цель

Поднять приложение как кликабельный каркас: канвас, layout, основные панели-заглушки, hitbox'ы зон.

### Что входит

- загрузка фонового изображения
- fixed desktop layout
- отображение основных панелей без полной бизнес-логики
- карта 7 зон и hitbox'ы
- клик по зоне как событие интерфейса

### Что PM режет на задачи

- shell приложения и базовый layout
- слой констант зон и координат
- визуализация hitbox / zone mapping
- базовые панели: control area, dashboard area, overlays buttons
- открытие pop-up по клику на зону

### Что можно тестировать после этапа

- приложение открывается и рендерит канвас
- все 7 зон кликабельны
- клик по каждой зоне открывает pop-up именно для этой локации
- layout не разваливается на целевом desktop viewport

### Результат этапа

Есть живая “карта фермы”, по которой уже можно кликать и на которую потом будет насаживаться доменная логика.

---

## Stage 2. Event log и ручной ввод

### Цель

Сделать append-only лог событий и ручное добавление событий через зону.

### Что входит

- модель события
- автоинкремент `id`
- manual input через zone popup
- фильтрация допустимых типов по матрице `location × event_type`
- запись событий в log
- отображение log list

### Что PM режет на задачи

- event model + constants
- reducer / append-only write path
- runtime guard совместимости
- pop-up ручного ввода
- log panel

### Что можно тестировать после этапа

- при клике по зоне можно вручную добавить валидное событие
- событие появляется в логе с корректными `id`, `location`, `event_type`, `intensity`
- недопустимые типы для зоны недоступны в UI
- записи лога не редактируются и не удаляются

### Результат этапа

У системы появляется первый настоящий источник данных: фермер вручную создаёт события, и они честно живут в append-only логе.

## Stage 3. Игровое время, seed и симулятор

### Цель

Запустить живую сессию: игровые часы, synthetic history, play/pause, fast-forward, автогенерация событий.

### Что входит

- игровые часы
- pause / resume
- fast-forward на 1 час
- seed истории при старте
- action “пересоздать историю”
- random simulator с валидными комбинациями

### Что PM режет на задачи

- clock state и now-window model
- seed generation
- simulator engine
- controls: run/pause/fast-forward/regenerate seed
- source tagging: `seed | sim | manual`

### Что можно тестировать после этапа

- при старте лог уже не пустой из-за seed
- simulator при запуске дописывает события в лог
- pause останавливает генерацию
- fast-forward меняет игровое время без генерации новых событий
- пересоздание seed меняет только `source=seed` часть истории

### Результат этапа

Приложение перестаёт быть статической формой и начинает жить как симуляция.

---

## Stage 4. Presence и priority по зонам

### Цель

Научить систему считать risk picture по локациям: `credibility`, `presence`, `priority`.

### Что входит

- вычисление `credibility`
- временное окно 1 час
- `presence` с override по `τ`
- `presence = 0` при пустом окне зоны
- `priority = presence × asset_value`
- сортировка зон и tie-breaker
- вывод per-zone метрик в dashboard

### Что PM режет на задачи

- projection helpers
- window filtering
- presence math
- priority math
- per-zone board / table

### Что можно тестировать после этапа

- после добавления/генерации событий видны `presence` и `priority` по зонам
- событие с `credibility >= τ` даёт `presence = 1.0`
- пустая зона получает `presence = 0`
- зоны ранжируются по `priority`
- fast-forward выкидывает старые события из окна и меняет значения

### Результат этапа

Система уже умеет отвечать на главный spatial-вопрос: где риск выше и почему.

---

## Stage 5. Оценка численности и BI-ядро

### Цель

Добавить второй смысловой слой: диапазон численности `low-high`, `pointEstimate`, `confidencePercent`, доказательства количества и top signals.

### Что входит

- `concurrency_window`
- нижняя и верхняя границы `low/high`
- `pointEstimate = low`
- `confidencePercent`
- блок “доказательство количества”
- блок “сильнейшие сигналы”

### Что PM режет на задачи

- projection численности
- formatter и explanation layer
- BI cards для estimate/confidence
- derived list для evidence/top-signals

### Что можно тестировать после этапа

- dashboard показывает `low-high`, точечную оценку и уверенность
- при событиях в нескольких зонах внутри окна одновременности растёт `low`
- при широкой активности растёт `high`
- пустое окно даёт `0%`
- BI показывает, какие именно события дали нижнюю границу

### Результат этапа

Система уже отвечает не только “где опасно”, но и “сколько кроликов предположительно есть”.

---

## Stage 6. Рекомендации и dog patrol

### Цель

Сделать систему интерактивной с точки зрения решения: рекомендации, тумблер пса, влияние действий фермера на модель.

### Что входит

- urgency level по `priority`
- текст рекомендации по доминирующему сигналу
- зона “нет активности”
- dog toggle
- suppression logic в simulator
- disabled `Следы` в Огороде при активном псе
- safety-net в reducer

### Что PM режет на задачи

- recommendation engine
- dominant signal logic
- dog state + UI toggle
- dog effect in simulator
- dog effect in manual input

### Что можно тестировать после этапа

- каждая активная зона получает рекомендацию нужного уровня
- при низком `priority` и `presence=1.0` показывается special note для малоценной зоны
- включение пса снижает огородные сигналы в симуляторе
- `Следы` в Огороде становятся disabled в manual input
- метрики и рекомендации реально меняются после dog toggle

### Результат этапа

Появляется замкнутый продуктовый loop: система оценивает риск, советует действие, действие меняет картину.

---

## Stage 7. Параметры estimator'а и объяснимость

### Цель

Дать пользователю возможность крутить модель и сразу видеть чувствительность расчётов.

### Что входит

- panel с `k`, `τ`, `concurrency_window`, priority thresholds, `dogSuppression`
- мгновенный пересчёт projection
- понятные подписи параметров
- сохранение логики “seed density не live-параметр projection”

### Что PM режет на задачи

- estimator settings state
- parameter inputs
- live recompute wiring
- copy / helper text around parameters

### Что можно тестировать после этапа

- изменение `τ`, `k`, `concurrency_window` меняет цифры на dashboard сразу
- изменение priority thresholds меняет уровни рекомендаций
- изменение `dogSuppression` меняет поведение огородных сигналов при активном псе
- seed density не пересчитывает projection сама по себе, а работает только через regenerate action

### Результат этапа

Пользователь уже может исследовать модель, а не только смотреть на фиксированную симуляцию.

## Stage 8. UX-полировка и финальный demo layer

### Цель

Довести проект до презентабельного demo-state: badge notifications, overlays, pixel-fit layout, AI Worklog / Legend.

### Что входит

- badge stack behavior
- animation polish
- overlay `AI Worklog`
- overlay `Legend`
- финальная подгонка панелей по канвасу
- косметическая доводка текста и spacing

### Что PM режет на задачи

- badge layer
- overlays
- content fill for static overlays
- layout polish / pixel fit

### Что можно тестировать после этапа

- новые события показывают badge в нужной зоне
- несколько событий в одной зоне дают стек `×N`
- открываются оба overlay
- финальный layout читаемый и не перекрывает критичные зоны

### Результат этапа

Это уже полноценный demo-артефакт, который можно показывать и руками гонять по сценарию.
