**ФЕРМА НЕВИДИМЫХ КРОЛИКОВ**

Тестовое задание · ТЗ v2.0

# 1. Обзор проекта

| | |
|---|---|
| Название | Ферма невидимых кроликов |
| Назначение | Интерактивная система оценки скрытой популяции кроликов на ферме по косвенным сигналам (датчики, следы, повреждения) и выдачи рекомендаций фермеру по защите урожая |
| Тип | Одностраничное приложение, без внешних сервисов и брокеров |
| Данные | Хранятся в памяти приложения на время сессии |

Пользователь (фермер) не видит кроликов напрямую. Система получает поток косвенных сигналов, оценивает вероятность присутствия кроликов по локациям, приоритет каждой локации с точки зрения риска для урожая, диапазонную оценку общей численности и выдаёт рекомендации к действию.

## 1.1 Что должно быть понятно из интерфейса (бриф)

1. Сколько кроликов предположительно на ферме.
2. Насколько система уверена в оценке.
3. Какие сигналы сильнее всего повлияли на вывод.
4. Какие рекомендации получает фермер.
5. Как меняется оценка, если поменять данные или параметры.

# 2. Архитектура

Паттерн: **event sourcing** в масштабе одного компонента, без событийно-ориентированной коммуникации между сервисами (EDA/брокеры не используются — приложение монолитное).

Состояние системы — не хранимая и перезаписываемая величина, а производная (projection), вычисляемая из неизменяемого лога событий. Лог только пополняется, существующие записи не редактируются и не удаляются.

Два источника событий пишут в один и тот же лог через **единую точку входа (reducer)**:
- **Симулятор** — генерирует случайные события по таймеру, пока не остановлен.
- **Форма ручного ввода** — активна только при остановленном симуляторе, добавляет пользовательские события.

Дополнительно при старте сессии лог **засевается синтетической историей** (см. 3.7).

Reducer только append'ит событие (тут же runtime-валидация по матрице 4.4). Projection (presence / priority / оценка численности / рекомендации) — **чистый мемоизированный селектор поверх лога**, никогда не пишется в стейт. Любое новое событие или изменение параметра немедленно триггерит пересчёт projection.

# 3. Доменная логика (core-слой, выполняется на клиенте)

## 3.1 Событие

Атомарная единица лога. Одно происшествие = одна запись.

| Поле | Тип | Описание |
|---|---|---|
| id | int | Автоинкремент. Единственный источник порядка событий — не зависит от `time` |
| event_type | enum | Тип сигнала (см. 4.2) |
| location | enum | Локация на ферме (см. 4.3) |
| intensity | int (1–10) | Выраженность сигнала. Учитывается для всех типов, включая следы |
| time | timestamp | Игровое время (см. 3.6). Метка для отображения и фильтрации по окну. Может дублироваться у разных событий |
| source | enum | `sim` \| `manual` \| `seed` — источник события (для логирования и отладки) |

## 3.2 Правдоподобие сигнала и Presence

### Правдоподобие (credibility)

Каждое событие имеет правдоподобие «это действительно кролик»:
```
credibility(событие) = confidence[event_type] × intensity/10   // для ВСЕХ типов, включая следы
```
`credibility ≥ τ` (порог, дефолт **τ = 0.5**) означает «гарантированно кролик».

**Следы не являются спецкейсом** — просто у них confidence = 1.0, поэтому они перешагивают τ легче всех (при intensity ≥ 5). Слабые следы (intensity < 5 → credibility < 0.5) гарантией не являются и идут как обычный сигнал. Гарантию может дать и не-след: пропажа моркови intensity=10 → 0.8 ≥ τ.

### Presence (уверенность в присутствии по локации)

Вычисляется по событиям локации во временном окне (см. 3.5):
```
если в окне по локации есть событие с credibility ≥ τ:
    presence = 1.0                                   // доказано, override
иначе:
    presence = avg(credibility) × (1 − e^(−n/k))     // среднее с насыщением по количеству
```
где `n` — число событий локации в окне.

- **Среднее, не сумма** — иначе много слабых сигналов обгоняет мало сильных, что не отражает реальную уверенность.
- **Множитель насыщения `(1 − e^(−n/k))`** чинит дыру «1 сигнал == 10 сигналов»: чистое среднее не учитывает количество подтверждений. `k` — коэффициент насыщения (меньше k → быстрее растёт доверие к количеству), дефолт **k = 1.5**. Множитель применяется только к ветке «иначе»; при override presence = 1.0 не считается.
- **Пустое окно по зоне (нет событий) → presence = 0** (не null): проще для priority (0 × asset_value = 0, зона внизу списка) и сортировки. «Слабая вероятность вместо уверенного нуля» — это про общий индекс численности (3.4), а не про presence отдельной зоны.

## 3.3 Priority (приоритет локации)

```
priority = presence × asset_value        // диапазон 0..10 при max asset_value = 10
```
- `asset_value` — ценность локации (см. 4.3), заданный вручную бизнес-параметр, не выводится из данных.
- Priority — итоговая величина ранжирования локаций и формирования рекомендаций.
- **Tie-breaker** при равном priority: presence desc → asset_value desc → `id` последнего события в зоне desc (свежесть) → порядок объявления локаций.

## 3.4 Оценка общей численности (диапазон + уверенность)

Оценка численности **отделена** от presence/priority: presence/priority отвечают за пространственный риск и рекомендации, оценка численности — за счёт голов. Обе математики делят только общий порог правдоподобия τ.

Принцип: **кролик не телепортируется**. Если в пределах окна одновременности `concurrency_window` (дефолт **5 сек**) сигналы пришли из K разных зон — на ферме минимум K кроликов.

- Зона считается **занятой** в момент T, только если внутри `concurrency_window` есть событие с `credibility ≥ τ`. Слабые сигналы (шум датчика, слабые следы) в счёт голов не идут, не раздувают оценку.
- **Нижняя граница** `low` = максимум за временное окно числа разных **занятых** зон одновременно.
- **Верхняя граница** `high` = число зон с любой активностью за окно (порог τ не применяется — верхняя граница максимально допускающая).
- **Точечная оценка** `pointEstimate = low` — честно «гарантированно не меньше low», без скрытых «подтягиваний вверх».
- **Уверенность**:
  ```
  confidencePercent = high === 0 ? 0 : Math.round((low / high) * 100)
  ```
  Только узость диапазона, без скрытых надбавок. `low=high` → 100%; `1..5` → 20%; чем уже диапазон, тем выше уверенность. Бонуса за presence=1.0 нет намеренно (следы уже влияют на `low`).
  **Пустое окно → 0%, НЕ 100%.** Отсутствие сигналов = слабая вероятность (кролики могли час не двигаться), а не «уверенный ноль». Пустое окно возможно только после fast-forward мимо всех данных (в норме окно наполнено за счёт seed, см. 3.7).

Оценка **сходится** с накоплением событий — fast-forward и симулятор двигают её к устойчивому значению, а не просто раздувают.

## 3.5 Временное окно

Оценка не потоковая — снэпшот по последнему **1 часу игрового времени**. События вне окна `[now − 3600s, now]` не участвуют в расчёте.

## 3.6 Игровое время и симулятор

### Игровые часы

- Тикают **1:1 реальному времени**, пока симулятор запущен; пауза замораживает часы.
- Ручной ввод берёт `time` из текущего замороженного значения часов.
- **Fast-forward** («промотать час вперёд») — мгновенно `now += 3600s` без генерации событий: старые события выпадают из окна, наглядно демонстрируя его работу.

### Симулятор

Простой генератор случайных событий на таймере (`setInterval` / рекурсивный `setTimeout`), один клиентский процесс, без внешних библиотек.
- Генерирует события только из допустимых комбинаций location × event_type (см. 4.4).
- intensity — случайное 1–10; распределение по локациям/типам равномерное по валидным комбинациям.
- Частота — одно событие раз в 2–5 сек (рандом), ~20 событий/мин.
- Работает только пока не на паузе; пауза не мутирует записанные события, только останавливает генерацию.
- Реализация — `Math.random()` + выбор из списка валидных комбинаций.

## 3.7 Seed стартовой истории

Ждать реальный час набора данных нереально, поэтому при старте сессии лог **засевается ~1 часом синтетической истории**:
- Тем же рандом-генератором (валидные комбинации 4.4, intensity 1–10), но `time` размазан по `[now − 3600s, now]`, `source = seed`.
- Seed-события — обычные записи append-only лога, неотличимы от живых, без спецобработки. Смысл: «ферма жила час до открытия дашборда».
- Окно «последний час» сразу наполнено → все метрики осмысленны с t=0.
- Число seed-событий = частота × 1ч (≈ несколько сотен), задаётся плотностью.
- **Плотность seed — параметр генерации, не projection.** Меняется только через действие «Пересоздать историю» (см. 5.1): сносит текущие `source=seed` события и генерит новый набор; live-события (sim/manual) не трогаются. Не пересчитывает projection напрямую — projection обновляется как следствие изменения лога. Батч-вставка через `SEED_BULK` (мимо anti-spam, см. 9.1).

## 3.8 Рекомендации фермеру

**Уровень срочности** — из порогов priority (уже location-aware через asset_value, отдельной ручной таблицы уровней по локациям НЕТ, иначе двойной учёт):

| Уровень | priority | Смысл |
|---|---|---|
| низкий | < 3 | наблюдать |
| средний | 3 … 6 | проверить / укрепить сектор |
| высокий | > 6 | срочно защитить урожай |

Пороги абсолютные (не относительные к максимуму зоны) — низкая ценность зоны честно тянет приоритет вниз даже при presence=1.0. Пороги настраиваемы в панели.

**Текст действия** — из справочника по доминирующему сигналу зоны (доминирующий = тип с макс. суммарным `credibility` в зоне за окно):

| Доминирующий сигнал / локация | Действие |
|---|---|
| Пропажа моркови — Огород | выпустить пса в огород |
| Пропажа моркови — Теплица | закрыть теплицу |
| Пропажа моркови — Сарай | забрать урожай с фермы |
| Новая яма — Огород / Теплица | осмотреть грядки, засыпать подкоп |
| Новая яма — Забор (любой сектор) | проверить / укрепить сектор [название] |
| Датчик движения — Теплица / Сарай / Забор | проверить зону лично |
| Следы — Огород / Теплица (high value) | подтверждено — приоритетная защита урожая |
| Следы — Забор (low value) | кролики подтверждены, урон маловероятен |
| Шуршание — Теплица / Сарай / Забор | проверить зону на месте |

- Итоговая рекомендация = уровень (из priority) + текст действия (из справочника).
- Невозможные по матрице 4.4 пары в справочник не входят.
- **Зона без активности** (presence = 0, нет событий в окне) → уровень «нет активности», действия нет (в списке внизу, без рекомендации).
- **Доминирующий сигнал при равной суммарной credibility** — tie-break: confidence типа desc (более достоверный тип побеждает) → `id` последнего события этого типа desc (свежесть).
- **Пометка при подтверждении в малоценной зоне:** если presence=1.0, но priority низкий из-за низкого asset_value — рекомендация явно отражает это («кролики подтверждены, но зона малоценна — защита урожая не требуется»), не молчит о подтверждённом присутствии.
- «Выпустить пса в огород» — не текст, а действие (тумблер), см. 3.9.

## 3.9 Пёс на ферме (dog patrol)

Активное контр-действие фермера, а не анимация. Состояние сессии `dogInGarden: boolean`, тумблер на дашборде. Рекомендация «выпустить пса в огород» подсвечивает этот тумблер.

Эффекты, пока пёс активен (точный алгоритм):
- **Симулятор** (веса выбора комбинаций, затем ренормализация):
  - `Следы × Огород` → вес **0** (полный запрет генерации — пёс не даёт появиться свежим следам).
  - Прочие комбинации Огорода → вес × `dogSuppression` (дефолт 0.2).
  - Все не-огородные комбинации → вес без изменений.
  Базовый выбор (пёс неактивен) — равномерный по валидным комбинациям.
- **Ручной ввод:** опция `Следы` в попапе Огорода **disabled** с подписью «пёс в огороде» (видна, но не выбираема — единый UX с фильтром матрицы, не «выбрал → упало на submit»). Остальные типы Огорода доступны. Reducer дополнительно отклоняет Следы×Огород при псе как safety-net (в лог, не в UI).
- Дашборд показывает индикатор «Пёс на ферме».

Демо-петля: пропажа моркови в Огороде → priority высокий → рекомендация «выпустить пса» → фермер жмёт тумблер → сигналы в Огороде падают → оценка едет вниз (наглядная чувствительность модели, требование №5 брифа).

## 4.3 Локации и их ценность (asset_value)

Плоский список равноправных сущностей. Забор разбит на секторы, каждый сектор — самостоятельная локация наравне с огородом/теплицей/сараем.

| Локация | asset_value | Комментарий |
|---|---|---|
| Огород | 10 | Без датчика движения — только визуальные сигналы (следы, пропажа моркови, новая яма) |

| Забор — Запад | 6 | Датчик движения |
| Забор — Юго-Запад | 6 | Датчик движения |
| Забор — Восток | 4 | Датчик движения |
| Забор — Юго-Восток | 2 | Датчик движения |

## 4.4 Совместимость типов событий и локаций

Не любой сигнал возможен в любой локации — ограничение обусловлено физической логикой (видимость следов, наличие датчиков, характер повреждений).

| Локация | Следы | Пропажа моркови | Новая яма | Шуршание | Датчик движения |
|---|---|---|---|---|---|
| Огород | ✅ | ✅ | ✅ | 🚫 | 🚫 |
| Теплица | ✅ | ✅ | ✅ | ✅ | ✅ |
| Сарай | 🚫 | ✅ | 🚫 | ✅ | ✅ |
| Забор (все секторы) | ✅ | 🚫 | ✅ | ✅ | ✅ |

Форма, симулятор и seed должны учитывать матрицу: недопустимые комбинации тип × локация не попадают в лог. Гарантируется runtime-guard'ом на входе в reducer (см. 8).

**Динамическое ограничение:** при активном псе (см. 3.9) пара **Следы × Огород** временно становится недопустимой (в ручном вводе → error-badge, в симуляторе не генерируется).

# 5. Фронтенд

## 5.1 Компоненты

- **Индикатор/управление симулятором** (запущен/остановлен).
- **Ручной ввод — клик по зоне** (активен только при остановленном симуляторе). Клик по хитбоксу зоны на канвасе (см. 5.6) открывает мини-попап у зоны с локацией **преднаполненной** из клика. Дропдаун локации не нужен — остаются тип события (отфильтрован матрицей 4.4 для этой зоны, с учётом динамического ограничения 3.9) + intensity (1–10). `id` и `time` проставляются автоматически (`time` = замороженные игровые часы, `id` = автоинкремент). Недопустимый выбор невозможен (список типов уже отфильтрован матрицей); при активном псе опция Следы в Огороде показана disabled с подписью «пёс в огороде» (3.9).
- **Лог событий** — список, без редактирования/удаления.
- **Оценка по локациям** — presence, priority, рекомендация.
- **Дашборд фермера (BI-плашка)** — см. 5.2.
- **Badge-уведомления** — см. 5.3.
- **Overlay «AI Worklog»** — статичный журнал разработки проекта (текст, скриншоты из excalidraw, забандленные ассеты; не редактируется в рантайме; без роутера — full-screen overlay по `useState`).
- **Overlay «Legend»** — статичный текст правил мира и допущений (пересказ матрицы 4.4 и §6 человеческим языком: «следов не бывает в сарае», «огород не шуршит» и т.п.).
- **Панель параметров estimator'а** — все спорные пороги/коэффициенты выведены инпутами, live-tunable, любое изменение мгновенно пересчитывает projection:
  - `k` — насыщение presence (дефолт 1.5)
  - `concurrency_window` — окно одновременности (дефолт 5 сек)
  - `τ` — порог правдоподобия «это кролик» (дефолт 0.5)
  - пороги priority (дефолт 3 и 6)
  - `dogSuppression` — множитель подавления сигналов в Огороде при активном псе (дефолт 0.2), см. 3.9
- **Действие «Пересоздать историю»** (не live-слайдер projection'а): плотность seed — параметр *генерации*, не projection. Кнопка сносит все `source=seed` события и генерит новый seed заданной плотности; live-события (sim/manual) сохраняются. См. 3.7.

## 5.2 Дашборд фермера (BI)

Аналитическая плашка внизу канваса (под интерактивной зоной фермы, в пределах фоновой картинки) — цифры и, возможно, простые графики поверх фона. Никакой подсветки зон. Делает очевидными пять вещей из брифа (§1.1):

1. **Сколько кроликов предположительно** — диапазон `low`–`high` + точечная оценка (см. 3.4).
2. **Уверенность %** — см. 3.4.
3. **Какие сигналы сильнее всего повлияли** — две раздельные группы:
   - *Доказательство количества* — набор одновременных событий, задавших нижнюю границу (напр. «12:03 — следы в Огороде + датчик ЮЗ + шуршание в Теплице → минимум 3 кролика»).
   - *Сильнейшие сигналы* — топ-3 события по `вклад = credibility(событие) × asset_value(зона)`, desc. Явная формула, без leave-one-out и пересчёта priority. Учитывает и силу сигнала, и ценность зоны.
4. **Рекомендации фермеру** — см. 3.8.
5. **Как меняется оценка при смене данных/параметров** — весь борд реактивен: ручной ввод, правка параметров панели и тумблер пса мгновенно двигают все цифры (демонстрация чувствительности модели).

Плюс **тумблер «Пёс на ферме»** + индикатор его состояния (см. 3.9).

**Источник событий** — только рандом-симулятор и ручная форма (+ seed при старте).

## 5.3 Badge-уведомления

Всплывающие уведомления о конкретных новых событиях в зоне («пропала морковка», «кто-то шуршит», «новая ямка»), появляются на пару секунд.
- **Якорь** — правый верхний угол хитбокса зоны (см. 5.6). Для забора — над соответствующим сектором-сенсором.
- При нескольких событиях в одной зоне за один момент — **стек со счётчиком** (badge «×3»), не replace/merge. Данные лога не трогаются (append-only).
- Уведомления — только об успешно добавленных событиях. «Пёс в огороде» — НЕ badge, а disabled-подпись опции Следы в попапе (3.9); в лог такое событие не пишется, badge не всплывает.
- Точная механика анимации (появление, длительность, поведение при плотных тиках) — подгоняется по картинке в реализации.

## 5.4 Поведение

- Любое новое событие (симулятор/форма/seed) немедленно триггерит пересчёт projection по всем локациям.
- Переключение симулятора не изменяет и не удаляет записи лога.

# 6. Допущения

- Огород, Теплица и Сарай имеют собственные датчики фиксации активности — сигнал закреплён за конкретной локацией. Огород — без датчика движения (только визуальные сигналы).
- Кролики не покидают ферму, не плодятся и не умирают — популяция N фиксирована в рамках сессии, но **перемещаются** между зонами. Поэтому «сумма presence по зонам» завышает N — счёт голов идёт по одновременности (см. 3.4).
- Система даёт **оценку-диапазон** и уверенность, а не точный подсчёт особей. Диапазон честно отражает неопределённость.
- Все сигналы, кроме перешагнувших τ, считаются независимыми свидетельствами (без учёта корреляции между типами).
- asset_value — заданный вручную бизнес-параметр, не производная от сигналов.
- Данные существуют только в рамках одной сессии, без персистентного хранилища.
- Интерфейс — desktop-only, фиксированная минимальная ширина, без адаптации под мобильные.

# 7. Риски

| Риск | Описание | Митигация |
|---|---|---|
| Искажение оценки при коротком окне | Узкое окно даёт нестабильную оценку при малом числе событий | Seed наполняет окно с t=0; уверенность падает при узости данных |
| Ложное чувство точности | Presence/численность — эвристика, не точный подсчёт | Оценка подаётся диапазоном + уверенность %; пустое окно = 0% уверенности, не «уверенный ноль» |
| Совпадение сигналов в одном тике | Несколько событий с одинаковым `time` не должны создавать неоднозначность порядка | Обеспечивается полем `id` |
| Раздувание N из шума | Слабые сигналы могли бы ложно поднимать нижнюю границу численности | Порог τ — в счёт голов идут только события с credibility ≥ τ |
| Недооценка при тихом присутствии | Зона с высокой intensity, но без сигналов ≥ τ, не достигает presence = 1.0 | Принятый компромисс; τ настраиваем в панели |

# 9. Нефункциональные требования

## 9.1 Anti-spam guard

Все действия, пишущие в лог или мутирующие часы (toggle симулятора, submit формы, fast-forward), защищены от программного/кликер-спама:
- **UI-слой:** кнопка блокируется на время обработки (`isProcessing`) — для нормального юзера.
- **Domain-слой (основная защита):** reducer — единственная точка записи — отклоняет действие, если с предыдущего прошло меньше `minInterval = 200ms`. UI-дизейбл обходится прямым dispatch, поэтому guard живёт в reducer.
- **Область действия — только пользовательские действия** (submit ручного ввода, toggle симулятора, fast-forward, toggle пса). НЕ под guard'ом: seed (батч-экшен `SEED_BULK` — вставляет массив за один dispatch при старте/пересоздании истории) и тик симулятора (интервал 2–5с и так > 200ms). Иначе seed из сотен событий конфликтует с 200ms-лимитом.

## 9.2 Логирование процессов в консоль

Структурированное namespaced-логирование ключевых шагов (`[reducer]`, `[projection]`, `[N-estimate]`, `[sim]`, `[seed]`):
- **append**: id, type, location, intensity, time, source; отклонение guard'ом (причина: невалидная комбинация / anti-spam).
- **projection**: триггер, размер лога, размер окна.
- **оценка численности**: low/high/estimate/confidence + разбор — какие зоны/события задали `low`, какие отсеклись порогом τ и почему.
- **presence/priority**: итоговые значения по зонам, сработал ли override.
- **рекомендации**: какое правило/порог выбрало каждый текст.
- **симулятор/seed**: старт/пауза, сгенерированное событие, число seed-событий.

Уровни `debug`/`info`/`warn`, глушится флагом. Инструмент разработки, не часть UI.
