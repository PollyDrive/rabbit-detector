# Roadmap выполнения: «Ферма невидимых кроликов» после Stage 2

Этот roadmap исходит из того, что **Stage 1 и Stage 2 уже сделаны**. Дальше проект ведётся не линейной цепочкой `3 -> 4 -> 5 -> 6 -> 7 -> 8`, а через короткий контрактный этап, затем **параллельную волну из 4 независимых блоков**, и только потом через этапы склейки.

Идея простая:

- Stage 1-2 дают базовый UI и append-only log
- Stage 3 фиксирует контракты, чтобы команды не блокировали друг друга
- Stage 4A-4D делаются параллельно и тестируются на своём уровне Scope
- Stage 5 и дальше склеивают готовые блоки в единый продукт

Этот документ не отменяет ТЗ. Источник истины по продукту остаётся `docs/rabbit-farm-tz.md`.

## Принципы этого roadmap

- сначала стабилизируем **контракты**, потом параллелим реализацию
- acceptance tests пишутся на **уровне Scope задачи**, а не через случайный полный `render(<App />)`
- domain- и projection-задачи тестируются напрямую, без обязательной зависимости от финального UI
- UI-задачи могут жить на mocks и fixtures до этапа склейки
- cross-cutting logic типа dog patrol идёт после того, как готовы независимые блоки

## Stage 1. Базовый каркас и карта фермы

### Статус

Сделано. Это базовая точка входа для всего дальнейшего roadmap.

### Цель

Поднять приложение как кликабельный каркас: канвас, layout, основные панели-заглушки, hitbox'ы зон.

### Что входит

- загрузка фонового изображения
- fixed desktop layout
- отображение основных панелей без полной бизнес-логики
- карта 7 зон и hitbox'ы
- клик по зоне как событие интерфейса

### Что можно тестировать после этапа

- приложение открывается и рендерит канвас
- все 7 зон кликабельны
- клик по каждой зоне открывает pop-up именно для этой локации
- layout не разваливается на целевом desktop viewport

### Результат этапа

Есть живая карта фермы, по которой уже можно кликать и на которую можно навешивать доменную логику.

## Stage 2. Event log и ручной ввод

### Статус

Сделано. Это доменная база для всех следующих этапов.

### Цель

Сделать append-only лог событий и ручное добавление событий через зону.

### Что входит

- модель события
- автоинкремент `id`
- manual input через zone popup
- фильтрация допустимых типов по матрице `location × event_type`
- запись событий в log
- отображение log list

### Что можно тестировать после этапа

- при клике по зоне можно вручную добавить валидное событие
- событие появляется в логе с корректными `id`, `location`, `event_type`, `intensity`
- недопустимые типы для зоны недоступны в UI
- записи лога не редактируются и не удаляются

### Результат этапа

У системы появляется первый настоящий источник данных: фермер вручную создаёт события, и они честно живут в append-only логе.

## Stage 3. Контрактный слой для параллельной разработки

### Цель

Зафиксировать общие контракты между runtime, projection и UI, чтобы следующие 4 блока можно было делать независимо.

### Что входит

- канонический shape `FarmEvent`, `FarmState` и `EstimatorSettings`
- единый список action / selector contracts
- shape projection-результатов для dashboard и per-zone board
- shared fixtures для событий и игровых сценариев
- test helpers для provider-level, selector-level и mocked-UI тестов
- правило: domain/projection acceptance не ходят через полный `App`, если их Scope уже и чище

### Что можно тестировать после этапа

- runtime-блок может писать события в лог с общим контрактом, не дожидаясь dashboard
- projection-блок может считать presence/priority/estimate по fixtures, не дожидаясь simulator
- UI-блок может рендерить dashboard и controls на mocked selector output
- все команды используют одну и ту же форму event fixtures и settings fixtures

### Результат этапа

Появляется стабильный шов между блоками. После него можно безопасно распараллелить разработку без ожидания полного предыдущего этапа.

---

## Stage 4A. Runtime и живые источники событий

### Цель

Сделать живую сессию, в которой лог заполняется автоматически и управляется игровым временем.

### Что входит

- игровые часы и `now-window model`
- pause / resume
- fast-forward на 1 час
- seed истории при старте
- action “пересоздать историю”
- simulator engine с валидными комбинациями
- source tagging: `seed | sim | manual`

### Что можно тестировать после этапа

- при старте лог уже не пустой из-за seed
- simulator при запуске дописывает события в лог
- pause останавливает генерацию
- fast-forward меняет игровое время без генерации новых событий
- пересоздание seed меняет только `source=seed` часть истории
- manual event получает текущее замороженное игровое время

### Результат этапа

Есть автономный runtime-слой, который честно кормит лог событиями и двигает игровое время, не завися от готовности аналитики и финального dashboard.

---

## Stage 4B. Presence и priority kernel

### Цель

Реализовать spatial-risk математику как чистый projection-слой поверх лога.

### Что входит

- вычисление `credibility`
- временное окно 1 час
- `presence` с override по `τ`
- `presence = 0` при пустом окне зоны
- `priority = presence × asset_value`
- сортировка зон и tie-breaker
- per-zone projection contracts для UI

### Что можно тестировать после этапа

- событие с `credibility >= τ` даёт `presence = 1.0`
- пустая зона получает `presence = 0`
- слабые сигналы идут по ветке среднего с насыщением, а не override
- зоны ранжируются по `priority`
- tie-breaker отрабатывает одинаковые `priority` предсказуемо
- fast-forward на уровне fixture-данных выкидывает старые события из окна и меняет значения

### Результат этапа

Появляется независимый projection-kernel, который умеет отвечать на вопрос “где риск выше и почему”, даже без готового simulator UI.

---

## Stage 4C. Оценка численности и recommendation engine

### Цель

Реализовать второй слой аналитики: сколько кроликов предположительно есть и что система советует делать.

### Что входит

- `concurrency_window`
- нижняя и верхняя границы `low/high`
- `pointEstimate = low`
- `confidencePercent`
- dominant signal logic
- urgency level по `priority`
- recommendation text по справочнику
- зона “нет активности”
- special note для малоценной зоны при `presence = 1.0`
- derived data для evidence и top signals

### Что можно тестировать после этапа

- при событиях в нескольких зонах внутри окна одновременности растёт `low`
- при широкой активности растёт `high`
- пустое окно даёт `0%`
- dashboard data показывает `low-high`, точечную оценку и уверенность
- каждая активная зона получает рекомендацию нужного уровня
- при низком `priority` и `presence = 1.0` показывается special note для малоценной зоны
- dominant signal выбирается предсказуемо при равной суммарной `credibility`

### Результат этапа

Появляется независимый аналитический слой, который отвечает не только “где опасно”, но и “сколько кроликов есть” и “что делать фермеру”.

---

## Stage 4D. Dashboard, controls и demo-UI scaffold

### Цель

Подготовить UI-слой так, чтобы он мог собираться и тестироваться на mocks до подключения реальных projection и runtime блоков.

### Что входит

- dashboard cards для per-zone board, estimate, confidence и recommendations
- control UI для simulator state, fast-forward и regenerate seed
- panel shell для estimator settings
- визуальные состояния для пустых данных, loading-like mocks и “нет активности”
- wiring на mocked provider / selector adapters
- layout-слоты под будущие badges и overlays

### Что можно тестировать после этапа

- dashboard рендерит per-zone данные из mocked selector output
- estimate / confidence / recommendations отображаются без реального simulator
- control UI шлёт правильные callback'и и не зависит от конкретной реализации reducer
- пустой dashboard читаем и не разваливает layout
- можно руками показать demo-shell на фикстурных данных, не дожидаясь склейки

### Результат этапа

Есть самостоятельный UI-слой, готовый к быстрой склейке с реальными runtime и projection блоками.

---

## Stage 5. Склейка runtime, projection и UI

### Цель

Подключить реальные источники данных и реальные projections к уже готовому dashboard, убрав временные mocks.

### Что входит

- wiring runtime state в общий provider
- подключение реальных selector'ов вместо mock adapters
- единый поток `log + now + settings -> projection -> UI`
- синхронизация controls с runtime
- устранение дублирующей временной логики из demo-shell

### Что можно тестировать после этапа

- simulator, manual input и seed реально меняют dashboard
- fast-forward сразу перестраивает per-zone board, estimate и recommendations
- один и тот же log используется и для spatial metrics, и для estimate, и для recommendation layer
- UI больше не зависит от моковых projection-данных

### Результат этапа

Четыре независимых блока превращаются в одну работающую систему без временных прокладок.

---

## Stage 6. Dog patrol и cross-cutting feedback loop

### Цель

Добавить первую настоящую cross-cutting механику, которая меняет и ввод, и симуляцию, и рекомендации, и итоговые метрики.

### Что входит

- `dogInGarden` state
- dog toggle в dashboard
- suppression logic в simulator
- disabled `Следы` в Огороде при активном псе
- reducer safety-net для `Следы × Огород`
- подсветка/индикатор состояния пса
- реакция recommendation layer на dog-related сценарии

### Что можно тестировать после этапа

- включение пса снижает огородные сигналы в симуляторе
- `Следы` в Огороде становятся disabled в manual input
- reducer отклоняет запрещённую комбинацию как safety-net
- метрики и рекомендации реально меняются после dog toggle
- рекомендация “выпустить пса” превращается в наблюдаемое изменение модели

### Результат этапа

Появляется замкнутый продуктовый loop: система оценивает риск, советует действие, действие меняет модель, и пользователь это видит.

## Stage 7. Параметры estimator'а и объяснимость

### Цель

Сделать модель исследуемой: пользователь меняет параметры и сразу видит, как это влияет на расчёты.

### Что входит

- panel с `k`, `τ`, `concurrency_window`, priority thresholds, `dogSuppression`
- live recompute wiring для projection
- helper text и понятные подписи параметров
- сохранение логики “seed density не live-параметр projection”
- объяснимые блоки evidence и top signals в финальном UI

### Что можно тестировать после этапа

- изменение `τ`, `k`, `concurrency_window` меняет цифры на dashboard сразу
- изменение priority thresholds меняет уровни рекомендаций
- изменение `dogSuppression` меняет поведение огородных сигналов при активном псе
- evidence/top-signals действительно объясняют текущий вывод
- seed density не пересчитывает projection сама по себе, а работает только через regenerate action

### Результат этапа

Пользователь уже может не просто смотреть на симуляцию, а исследовать чувствительность модели и понимать, почему она пришла к конкретному выводу.

## Кто и как параллелит

### Роль PM

- после готового Stage 2 PM поднимает **Stage 3 как короткий контрактный этап**
- после готового Stage 3 PM режет **четыре независимые группы issue внутри параллельной волны**
- у каждой группы свой Scope, свои acceptance tests и своя test-ветка
- PM не пишет acceptance “на весь App” для задач, которые живут на уровне reducer, selector или mocked UI

Рекомендуемые группы задач:

- `runtime` — Stage 4A
- `spatial-projection` — Stage 4B
- `estimate-recommendations` — Stage 4C
- `dashboard-ui` — Stage 4D

### Роль implementer

- один implementer берёт только одну группу задач за раз
- implementer в `runtime` не ждёт готовый dashboard
- implementer в `spatial-projection` и `estimate-recommendations` работает через pure functions и fixtures
- implementer в `dashboard-ui` работает через mocked provider / selector adapters
- implementer, который берёт Stage 5 и выше, уже занимается склейкой, а не изолированным блоком

### Роль reviewer

- reviewer проверяет, что задача тестируется на своём уровне абстракции
- reviewer отклоняет acceptance, который случайно зависит от соседней группы задач
- reviewer отдельно следит, чтобы до Stage 5 никто не тащил лишнюю интеграцию в domain-only PR

### Роль integration owner

- после того как 4A-4D готовы, один владелец склейки берёт Stage 5
- этот же человек или отдельный implementer берёт Stage 6, потому что dog patrol уже реально кросс-срезовый
- Stage 7 и 8 можно снова частично распараллелить: один человек ведёт explainability/settings, другой — polish/overlays/badges

### Как открывать ветки и PR

- Stage 3 идёт как обычный короткий этап
- после него PM может открыть несколько `_test`-веток одного product-wave, если acceptance действительно независимы
- все ветки режутся от свежего `preprod`
- каждая issue остаётся 1:1 с implementer-веткой и implementer-PR
- сквозные full-App acceptance держим в Stage 5 и выше, а не размазываем по ранним параллельным блокам

### Что нельзя делать

- нельзя писать domain acceptance через UI, если Scope задачи уже и чище
- нельзя требовать от Stage 4B или 4C готового simulator UI
- нельзя требовать от Stage 4D готовой математики, если контракт projection уже зафиксирован
- нельзя начинать cross-cutting dog patrol до того, как готовы runtime, projection и dashboard wiring
